# 项目 001: AI 每日故事 - 详细方案

## 📖 项目概述

**项目名称**: AI 每日故事 (AI Daily Story)

**一句话介绍**: AI 每天生成一个专属互动故事，你的选择改变剧情

**推荐优先级**: Top 1

**核心价值**: 通过 AI 生成无限可能的互动故事，结合每日更新的稀缺性机制，打造用户日常习惯和社交传播。

---

## 🎮 详细玩法说明

### 核心流程

1. **每日登录奖励**
   - 用户每天首次登录获得一个全新的 AI 生成故事开头
   - 故事类型随机或根据用户偏好选择（悬疑/爱情/冒险/科幻等）
   - 未阅读的昨日故事自动归档，可继续但不再计入"每日"统计

2. **互动选择系统**
   - 故事在关键节点暂停，提供 2-4 个选择
   - 每个选择导致不同的剧情分支
   - 选择后 AI 即时生成后续剧情（约 200-500 字）
   - 支持"自定义行动"输入框，允许自由输入

3. **剧情树系统**
   - 可视化剧情分支树，展示已探索路径
   - 支持回溯到之前的选择点重新选择
   - 统计探索度（已探索分支/总分支）

4. **社交分享**
   - 生成故事卡片：包含关键剧情和选择路径
   - 分享"我的今日故事"到社交媒体
   - 好友可以查看你的剧情路径并评论
   - "剧情挑战"：好友基于你的开头继续创作

### 进阶玩法

- **连续签到奖励**: 连续 7 天解锁特殊故事主题
- **成就系统**: "探索 100 个分支"、"达成隐藏结局"等
- **主题周活动**: 每周一个特定主题（恐怖周、爱情周等）
- **用户创作**: 付费用户可以创作自己的故事模板供他人体验

### 用户旅程示例

```
Day 1: 登录 → 获得悬疑故事开头 → 做出 3 个选择 → 达成结局 A → 分享
Day 2: 登录 → 获得爱情故事开头 → 做出 5 个选择 → 达成结局 B → 收藏
Day 3: 登录 → 获得昨日故事续集机会 → 选择继续或新故事 → ...
```

### 技术栈选型

| 层级 | 技术选型 | 理由 |
|------|----------|------|
| 前端 | 微信小程序 + Taro | 快速开发，跨平台，社交传播便利 |
| 后端 | Node.js + NestJS | 高并发，TypeScript，生态丰富 |
| 数据库 | PostgreSQL + Redis | 关系型数据 + 缓存，成熟稳定 |
| AI | DashScope (通义千问) | 中文优化，成本可控，国内访问快 |
| 部署 | 阿里云 ECS + Docker | 弹性扩容，运维简单 |
| 监控 | Prometheus + Grafana | 开源成熟，可视化好 |

### 核心 API 设计

```typescript
// 1. 获取今日故事
GET /api/v1/story/daily
Response: {
  storyId: string,
  title: string,
  genre: string,
  content: string,
  choices: Array<{id, text, description}>
}

// 2. 提交选择，获取后续剧情
POST /api/v1/story/choice
Body: { storyId, choiceId, customAction? }
Response: {
  nextContent: string,
  newChoices: Array,
  isEnding: boolean,
  endingType?: string
}

// 3. 获取剧情树
GET /api/v1/story/:id/tree
Response: {
  nodes: Array<{id, title, visited, current}>,
  edges: Array<{from, to}>,
  progress: number
}

// 4. 分享故事卡片
GET /api/v1/story/:id/share-card
Response: { imageUrl, summary, path }
```

### 数据库设计

```sql
-- 用户表
CREATE TABLE users (
  id VARCHAR(32) PRIMARY KEY,
  nickname VARCHAR(50),
  avatar_url VARCHAR(255),
  created_at TIMESTAMP,
  last_login DATE
);

-- 故事表
CREATE TABLE stories (
  id VARCHAR(32) PRIMARY KEY,
  user_id VARCHAR(32),
  title VARCHAR(100),
  genre VARCHAR(20),
  status VARCHAR(20), -- ongoing, completed, archived
  created_at DATE,
  updated_at TIMESTAMP
);

-- 剧情节点表
CREATE TABLE story_nodes (
  id VARCHAR(32) PRIMARY KEY,
  story_id VARCHAR(32),
  parent_node_id VARCHAR(32),
  content TEXT,
  choices JSONB, -- 存储选择项
  is_ending BOOLEAN,
  ending_type VARCHAR(50)
);

-- 用户进度表
CREATE TABLE user_progress (
  user_id VARCHAR(32),
  story_id VARCHAR(32),
  current_node_id VARCHAR(32),
  visited_nodes JSONB,
  completed_at TIMESTAMP,
  PRIMARY KEY (user_id, story_id)
);
```

### AI Prompt 设计

```
系统角色：你是一位专业的互动小说作家，擅长创作引人入胜的故事。

任务：根据以下信息生成故事内容：
- 故事类型：{genre}
- 前情提要：{previous_context}
- 用户选择：{user_choice}

要求：
1. 字数控制在 200-400 字
2. 在关键处设置 2-4 个有意义的选择
3. 保持故事连贯性和悬念
4. 避免敏感内容
5. 每个选择应有不同的风险/回报

输出格式：
{
  "content": "故事正文...",
  "choices": [
    {"id": "A", "text": "选择文本", "description": "简短说明"},
    ...
  ],
  "is_ending": false,
  "ending_type": null
}
```

## ⚠️ 风险评估与应对

### 技术风险

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|----------|

| API 成本失控 | 中 | 高 | 1. 每日调用限制 2. 免费用户限次 3. 监控告警 |
| 内容审核漏判 | 低 | 高 | 1. 双重审核机制 2. 用户举报 3. 关键词过滤 |
| 服务器宕机 | 低 | 高 | 1. 多可用区部署 2. 自动扩容 3. 数据备份 |

### 业务风险

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|----------|
| 用户留存低 | 高 | 高 | 1. 每日签到激励 2. 剧情连续性 3. 社交绑定 |
| 内容同质化 | 中 | 中 | 1. 多样化 Prompt 2. 主题轮换 3. 用户创作 |
| 变现困难 | 中 | 中 | 1. 免费增值模式 2. 广告变现 3. IP 授权 |
| 竞品模仿 | 高 | 中 | 1. 快速迭代 2. 品牌建设 3. 用户社区 |

### 合规风险

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|----------|
| 内容违规 | 中 | 高 | 1. AI 内容过滤 2. 人工抽检 3. 举报机制 |
| 数据隐私 | 低 | 高 | 1. 最小化收集 2. 加密存储 3. 隐私政策 |
| 版号问题 | 低 | 中 | 1. 以工具名义上线 2. 避免付费抽卡 3. 咨询法务 |

### 风险应对优先级

```
高优先级（立即处理）:

中优先级（上线前完成）:
├── 服务器高可用部署

低优先级（持续优化）:
```
